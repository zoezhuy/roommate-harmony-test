# Roommate Harmony Test｜AI 交互产品设计

**HCI / UX｜可解释评分系统｜Persona 映射｜前端交互原型**

## 项目简介
Roommate Harmony Test 是一个围绕“室友匹配”场景设计的 AI 交互产品原型，目标是帮助用户在合住前更早识别潜在生活摩擦点，并把原本难以启齿的现实问题，转化为更容易开始的结构化对话。

与传统只根据兴趣、专业或表面偏好进行匹配的方式不同，这个项目更关注真实共同生活中更容易引发冲突的核心因素，例如作息、清洁习惯、社交边界、噪音容忍度与沟通方式。项目通过多步问卷、行为维度建模、匹配算法和 Persona 映射，为用户提供一个更可解释、更具互动感、也更容易被年轻人接受的室友兼容性判断工具。

## 我的角色
在这个项目中，我主要负责：

- 设计完整用户流程：首页 → 问卷 → 结果页
- 定义关键产品指标：完成率、跳出率、用时
- 设计行为维度与评分逻辑
- 构建 6 维加权 Likert 用户模型
- 实现 0–100 匹配算法
- 通过 Persona 映射生成个性化解释
- 通过前端原型验证流程可行性

## 项目背景
很多室友匹配工具更关注“你们兴趣是否相似”，但真正决定共同居住体验的，往往不是表面的兴趣，而是更具体的生活方式差异，例如：

- 睡眠与作息是否一致
- 清洁与整理标准是否相近
- 是否接受访客与社交频率
- 对噪音和私人空间的容忍度
- 出现冲突时的沟通方式

这些问题通常不是“没有答案”，而是“在入住前没有被有效问出来”。

## 核心问题
这个项目试图解决的问题是：

**如何通过一个低压力、可解释、具备互动感的产品流程，帮助用户在入住前更早识别室友之间潜在的生活摩擦点？**

## 产品目标
- 帮助用户在合住前更早发现潜在冲突，而不是入住后才暴露问题
- 将“合不合适”从模糊感受转化为结构化反馈
- 提供可解释结果，而不是黑箱式匹配
- 让结果不仅有分数，还有可讨论的话题与建议
- 用更年轻化、更有亲和力的表达方式降低用户接受门槛

## 解决方案
Roommate Harmony Test 通过三层设计来完成这个目标：

### 1. 结构化问卷输入
通过多步问卷采集用户在不同生活维度上的偏好与习惯，包括：

- 作息与睡眠
- 清洁与整理
- 社交与访客
- 噪音与私人空间
- 规则感与沟通方式
- 日常生活习惯

### 2. 可解释评分引擎
项目基于 6 维加权 Likert 用户模型，将用户回答转化为结构化行为画像，并生成 0–100 的兼容性分数。

相比“只给结果”的测试工具，这套逻辑更强调：

- 分数来源透明
- 差异点可被解释
- 用户能知道潜在冲突可能出现在哪里
- 结果不仅用于判断，也用于支持沟通

### 3. Persona 映射与结果解释
为了降低用户面对“被评价”时的不适感，结果页没有直接使用生硬标签，而是通过 Persona 映射的方式，帮助用户理解自己的生活风格类型以及与他人可能的匹配关系。

这种设计让结果更容易被接受，也更适合在真实社交关系中被讨论。

## 为什么采用动物拟人画像
为了让测试结果更容易被年轻用户接受，这个项目借鉴了类似 MBTI 的维度化表达思路：不是直接给出冷冰冰的评价，而是通过用户在不同题目中的选择，识别其在作息、清洁、社交边界、沟通方式等维度上的倾向，并进一步映射为不同的**动物拟人 Persona**。

这种设计的价值在于：

- 比起直接评价“你适不适合当室友”，动物拟人画像的表达方式更轻松、更容易引发兴趣
- 相比单纯的匹配分数，Persona 更容易帮助用户理解自己的生活风格
- 对年轻用户来说，这种带有一点“人格测试”感的交互形式更熟悉，也更容易接受、分享和讨论
- 结果页不再只是给出一个答案，而是帮助用户用更低压力的方式开启关于现实合住问题的对话

因此，这个项目并不是直接套用 MBTI，而是**借鉴其“多维度归类 + 形象化解释”的产品表达方式**，让原本可能显得敏感、枯燥甚至带有评判感的匹配结果，变得更有亲和力、更易理解，也更符合年轻用户的使用习惯。

## 关键设计亮点

### 生活习惯优先，而不是兴趣匹配
这个项目的核心判断不是“你们喜不喜欢同样的东西”，而是“你们能不能舒服地住在一起”。

### 可解释，而不是黑箱
用户不仅能看到最终分数，也能理解影响结果的主要行为差异。

### 结果可行动，而不是只展示
结果页不仅告诉用户“合不合”，还尝试指出：

- 哪些地方最容易产生摩擦
- 哪些问题值得提前沟通
- 哪些生活规则可能帮助减少冲突

### 借鉴 MBTI 式表达，降低理解门槛
通过动物拟人 Persona 的方式呈现结果，让系统反馈更轻、更年轻化，也更容易被接受。

### 兼顾产品判断力与社交温度
这个项目不把结果做成绝对结论，而是把结果设计成“沟通起点”，减少标签化风险。
